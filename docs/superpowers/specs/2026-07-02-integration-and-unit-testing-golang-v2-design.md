# Design: integration-and-unit-testing-golang-v2 — Unit & Integration Testing Template

**Date:** 2026-07-02
**Status:** Approved (pending written-spec review)
**Author:** Sorrawich Yooboon (with Claude Code)

## 1. Purpose

A complete, reusable **prototype/template** for structuring a Go service and writing
**unit tests and integration tests**. The structure and testing conventions are modeled
1:1 on the reference repository `KKP/github/istd/istd-backend` (hexagonal architecture,
Ginkgo/Gomega, gomock, sqlboiler, oapi-codegen), with the integration-testing layer —
which is stale/incomplete in the reference — rebuilt to actually work end-to-end.

**Non-goal:** real business logic. Every handler/service/repository contains only the
minimum wiring needed to exercise its layer and demonstrate the corresponding test type.
The example domain (Order & Inventory) is a scaffold, nothing more.

## 2. Reference conventions carried over 1:1

From `istd-backend`:

| Concern | Convention |
|---|---|
| Architecture | Hexagonal / ports-and-adapters. Inward-pointing deps. Composition root in `internal/server`. |
| HTTP | Echo v4, handlers generated by oapi-codegen from an OpenAPI spec. |
| DB models | sqlboiler-generated models; hand-written `Repository` maps generated models ↔ domain entity. |
| Migrations | dbmate. |
| Unit tests | Ginkgo v2 + Gomega BDD (`Describe/Context/It`), **external test packages** (`pkg_test`), one `*_suite_test.go` per package, `ENV=unittest` + `config.Init(BasePath)` in `BeforeEach`. |
| Mocks | `go.uber.org/mock` (gomock) via `//go:generate mockgen -source=$GOFILE -destination=mock/$GOFILE`; mocks in a `mock/` subfolder beside each interface; fresh `gomock.NewController` per spec. |
| Repository tests | `DATA-DOG/go-sqlmock` asserting the exact SQL sqlboiler emits; shared helpers in `internal/utils/testing/sqlmock.go`. |
| Port (HTTP) tests | Boot a full `server.Application` with all handlers mocked; drive a test Echo context. |
| Errors | `errs/` AppError catalog + HTTP-status mapping; `internal/domains/errortype/` wrapped adapter errors. |
| Utilities | `pkg/*` pure helpers, each with its own specs. |
| Config | Typed `Config` struct; env-selected JSON (`config.<ENV>.json`); global singleton `config.Current`. |
| Tooling | `Makefile` (`unit-test`, `integration-test`, `lint`, `generate`, `gen-db`, `tools-install`), `sqlboiler.toml`, `.air.toml`, `.pre-commit-config.yaml`, `third_party/oapi-codegen/`. |

## 3. Deliberate improvements over the reference

The reference left two things stale; because this repo's *purpose* is testing, we fix them:

1. **Real integration suite.** `tests/ivt` is a working, hermetic **testcontainers-go**
   suite (spins Postgres + Kafka in Docker per run), gated by `//go:build integration`,
   organized as a Ginkgo suite. Replaces the reference's empty `tests/ivt` and its
   `//go:build integration` stdlib tests that hit hard-coded network endpoints.
2. **Real CI.** `.github/workflows/ci.yaml` runs `lint` + `unit-test` + `integration-test`
   (the reference had only a CodeQL workflow).

## 4. Additional requirement — two unit-test styles side by side

Beyond the Ginkgo BDD specs (primary, matching the reference), the template also
demonstrates the classic **table-driven test** style (standard library `testing`:
`func TestXxx(t *testing.T)` + `tests := []struct{...}` + `for` loop + `t.Run`), using
**testify** (`assert`/`require`) for assertions.

- **Scope:** representative per layer — one command, one query, one repository, one pkg
  utility (NOT every component; Ginkgo specs already cover everything).
- **Placement:** a `_table_test.go` file beside the Ginkgo `_test.go` for the same
  subject, so the two styles can be compared directly.
- **Header note** on each table-driven file, e.g.:
  ```go
  // Style: table-driven unit test (standard library "testing" + testify).
  // See create_order_test.go for the Ginkgo/Gomega BDD equivalent of the same subject.
  ```
- **Mocks are shared:** both styles use the same gomock mocks (gomock works with both
  `*testing.T` and `GinkgoT()`), so no mock is generated twice.

Net result: the template teaches **Gomega** (BDD) and **testify** (table-driven)
assertions, and **gomock** as the single mocking library.

## 6. Example domain — Order & Inventory (scaffold only)

| Flow | Trigger | Layers exercised |
|---|---|---|
| **CreateOrder** | `POST /orders` | REST port → `commands.CreateOrder` → order repo (INSERT) → Kafka produce `order.created` |
| **GetOrder** | `GET /orders/:id` | REST port → `queries.GetOrder` → cache → order repo (SELECT) |
| **ReserveStock** | consume `order.created` | Kafka consumer port → `commands.ReserveStock` → inventory repo (UPDATE) |
| **Health** | `GET /healthz` | REST port only |

Two entities (`order`, `inventory`); sync REST + async Kafka; DB + in-memory cache;
Kafka producer + consumer. This set is the minimum that touches every layer and every
test type.

## 8. Test matrix

| Test kind | Location | Tech | Demonstrates |
|---|---|---|---|
| Unit — command (BDD) | `services/commands/create_order_test.go` | Ginkgo + gomock | mock repo + kafka producer, Success/Error contexts |
| Unit — command (table) | `services/commands/create_order_table_test.go` | testing + testify + gomock | same subject, table-driven |
| Unit — query (BDD) | `services/queries/get_order_test.go` | Ginkgo + gomock | cache-hit / miss / db-error branches |
| Unit — query (table) | `services/queries/get_order_table_test.go` | testing + testify + gomock | same subject, table-driven |
| Unit — repository (BDD) | `domains/order/repository_test.go` | Ginkgo + go-sqlmock | asserts exact SQL |
| Unit — repository (table) | `domains/order/repository_table_test.go` | testing + testify + go-sqlmock | same subject, table-driven |
| Unit — REST port | `ports/restapis/order_create_test.go` | Ginkgo | mocked Application, test Echo ctx, status/body |
| Unit — consumer | `ports/consumers/order_consumer_test.go` | Ginkgo + gomock | feed message, assert command called |
| Unit — pkg util (BDD) | `pkg/retry/retry_test.go` | Ginkgo | pure-function specs |
| Unit — pkg util (table) | `pkg/retry/retry_table_test.go` | testing + testify | same subject, table-driven |
| Integration | `tests/ivt/*` | testcontainers-go + `//go:build integration` | real Postgres+Kafka: POST→row + event; publish event→inventory updated |

## 9. Tooling / Makefile targets

- `unit-test` → `ENV=unittest ginkgo -r -cover --skip-package tests/ivt,internal/domains/db_models_gen`
- `integration-test` → `ENV=integration ginkgo -r -tags=integration ./tests/ivt`
- `test` → `go test ./...` (runs Ginkgo suites + stdlib table-driven; excludes integration via build tag)
- `lint` → `golangci-lint run`
- `generate` → oapi-codegen → sqlboiler → `go generate ./...` (mocks) → `go mod tidy`
- `gen-db` → `sqlboiler psql`
- `migrate` → `dbmate up`
- `tools-install` → `go install` of oapi-codegen, mockgen, ginkgo, sqlboiler(+psql), dbmate

## 10. Config & errors

- **Config:** typed `Config` struct read from `configs/config.<ENV>.json` (viper). Global
  `config.Current`; `config.Init(opts...)` with a `BasePath` option so tests point at
  `../../../configs`. ENVs: default, `unittest`, `integration`.
- **Errors:** `errs.AppError{Code, Message, ...}` catalog with prefixes `ORD-E-4xxx`
  (validation/HTTP), `ORD-E-5xxx` (system), `ORD-E-6xxx` (business); `error_response.go`
  maps codes → HTTP status. `internal/domains/errortype/` holds wrapped adapter errors
  that services translate into `AppError`.

## 11. Out of scope (YAGNI)

- Real business rules, auth, multi-schema DBs, SOAP clients, Redis, S3, Prometheus dashboards.
- `cron` and `workerpool` layers (can be added later following the same pattern).
- Publishing the codegen'd client as a reusable `pkg/` SDK.

## 12. Open risks / notes

- **Docker required** for `integration-test` (testcontainers) and for `sqlboiler`/`generate`
  (needs a live Postgres to introspect the schema). Unit tests need only Go.
- `generate` requires the codegen tools installed (`make tools-install`).
- sqlboiler model generation depends on migrations being applied to a throwaway Postgres
  first; the `gen-db` flow will document/automate this (dbmate up against a Docker Postgres,
  then sqlboiler introspect).
